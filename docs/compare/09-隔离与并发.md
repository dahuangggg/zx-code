## 一、问题引入

当多个 agent 同时工作时，最怕的是什么？**文件冲突。**

```
Agent A: 正在修改 src/app.ts，添加认证模块
Agent B: 同时在修改 src/app.ts，添加日志模块
→ 谁的修改先写入？谁的被覆盖？
```

learn-claude-code 用 Git worktree 解决这个问题：

```python
# s18 的核心思想：每个任务一个独立的 git 工作目录

def create_worktree(task_id: str) -> str:
    branch = f"wt/{task_id}"
    path = f".worktrees/{task_id}"
    
    # git worktree = 同一个 repo 的独立工作目录
    # 各自有自己的文件系统，互不干扰
    run(f"git worktree add -b {branch} {path}")
    
    return path  # agent 在这个路径下工作

# 控制平面 (.tasks/) 和执行平面 (.worktrees/) 分离
# .tasks/task-001.json → 记录任务状态 + 关联的 worktree
# .worktrees/task-001/ → 独立的文件系统
```

## 二、三个仓库的隔离策略

### 2.1 Codex — OS 沙箱 + 进程隔离

```
Codex 的隔离是多层的:

层 1: OS 沙箱（最强隔离）
  → 每个工具调用都在沙箱里执行
  → 文件系统访问被限制（只能写 writable_roots）
  → 网络访问被限制（只能访问 allowed_domains）
  → 即使 agent 恶意/出错，也逃不出沙箱

层 2: 进程隔离（子代理）
  → 每个子代理是独立的 Codex 实例
  → 有自己的 session、history、context
  → 通过事件通道和父代理通信
  → 但共享同一个文件系统（沙箱范围内）

层 3: 会话隔离
  → SQLite 存储保证会话历史不混乱
  → 每个 session 有独立的 ID 和 context

注意: Codex 没有 git worktree 隔离！
原因: Codex 的设计更偏向"单任务执行"
多代理同时改代码的场景较少
沙箱已经提供了足够的安全保证
```

### 2.2 Pi-Mono — 会话树 + 分支

```typescript
// Pi-Mono 用会话树（session tree）实现逻辑隔离:

// 会话存储为 JSONL 文件，支持树形分支结构
// session.jsonl:
// {"id": "msg-1", "parentId": null, "type": "user", "content": "..."}
// {"id": "msg-2", "parentId": "msg-1", "type": "assistant", "content": "..."}
// {"id": "msg-3", "parentId": "msg-1", "type": "user", "content": "..."} ← 分支！
//
// msg-1 → msg-2 (分支 A)
//       → msg-3 (分支 B)

// /fork 命令创建新分支:
// → 当前对话在一个分支继续
// → 新分支从某个消息点开始
// → 两个分支的文件操作可能冲突！

// Pi-Mono 的解决方案:
// 1. 分支切换时做 branch summarization
//    → 离开的分支被压缩为摘要
// 2. 依赖用户自觉
//    → 不要同时在两个终端改同一个文件
// 3. 无 git worktree 支持

// 优点: 轻量，不需要 git
// 缺点: 多分支同时工作时没有文件隔离
```

### 2.3 Claude Code — Git Worktree 隔离（和 learn-claude-code 最像）

```typescript
// Claude Code 的 AgentTool 支持 worktree 隔离:

const result = await AgentTool.execute({
    description: "重构支付模块",
    prompt: "在独立分支上重构支付模块...",
    isolation: "worktree",  // ← 关键参数
});

// 内部实现:
async function createWorktreeForAgent(agentId: string) {
    // 1. 创建 git worktree
    const branchName = `agent/${agentId}`;
    const worktreePath = `.worktrees/${agentId}`;
    await exec(`git worktree add -b ${branchName} ${worktreePath}`);
    
    // 2. 子代理在 worktree 路径下工作
    // → 所有文件操作都在独立目录
    // → 不影响主工作目录
    
    // 3. 工作完成后
    if (agentMadeChanges) {
        // 返回 worktree 路径和分支名
        return { path: worktreePath, branch: branchName };
        // → 用户决定是否合并
    } else {
        // 没有改动，自动清理
        await exec(`git worktree remove ${worktreePath}`);
    }
}

// 和 learn-claude-code s18 的区别:
// s18: 控制平面(.tasks/) + 执行平面(.worktrees/) + 事件流
// Claude Code: 更简化——只有 worktree 创建/清理
// 没有 learn-claude-code 那么完整的事件流和双向绑定
```

## 三、并发执行策略对比

除了文件隔离，还有**工具执行并发**的问题：

```
模型一次返回 5 个工具调用:
  1. read_file("src/app.ts")
  2. read_file("src/utils.ts")
  3. grep("TODO", "src/")
  4. bash("npm test")
  5. edit_file("src/config.ts", ...)

哪些可以并行？哪些必须串行？
```

### 三个仓库的并发策略

```
┌────────────────────────────────────────────────────────────────┐
│ Codex: Orchestrator 单个执行                                    │
│                                                                │
│ Codex 的 ToolOrchestrator 默认逐个执行工具                     │
│ → 简单安全，不需要考虑并发问题                                 │
│ → 但速度较慢（5 个工具要串行 5 次）                            │
│                                                                │
│ 多代理场景下:                                                   │
│ → 不同 agent 并行（每个 agent 是独立的 tokio task）             │
│ → 同一个 agent 内的工具串行                                    │
└────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────┐
│ Pi-Mono: 显式的 parallel/sequential 模式                        │
│                                                                │
│ // 配置级别的选择:                                              │
│ const agent = new Agent({                                      │
│     toolExecutionMode: "parallel",  // 或 "sequential"          │
│ });                                                            │
│                                                                │
│ // parallel 模式:                                               │
│ // 1. 串行做 preflight 验证（确保参数合法）                    │
│ // 2. 并发执行所有工具 (Promise.all)                            │
│ // → 简单粗暴：要么全并行，要么全串行                          │
│ // → 不区分"安全"和"不安全"的工具                              │
│                                                                │
│ // sequential 模式:                                             │
│ // 逐个执行                                                    │
└────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────┐
│ Claude Code: 智能分批（最精细）                                  │
│                                                                │
│ StreamingToolExecutor 的分批逻辑:                               │
│                                                                │
│ 1. 对每个工具调用 isConcurrencySafe():                          │
│                                                                │
│    read_file("a.ts")     → safe ✓  (只读)                      │
│    read_file("b.ts")     → safe ✓  (只读)                      │
│    grep("TODO")          → safe ✓  (只读)                      │
│    bash("npm test")      → unsafe ✗ (可能有副作用)              │
│    edit_file("c.ts")     → safe ✓  (没有冲突的编辑)            │
│                                                                │
│ 2. safe 组 → Promise.all 并行（最多 10 并发）                   │
│    unsafe 组 → 逐个串行                                        │
│                                                                │
│ 3. 动态安全判断:                                                │
│    bash("ls")            → safe (无副作用)                      │
│    bash("echo hello")    → safe (无副作用)                      │
│    bash("rm file && ls") → unsafe (有 && 和副作用)              │
│    bash("curl url | jq") → unsafe (有管道和网络)                │
│                                                                │
│    edit_file("a.ts")     → safe                                 │
│    edit_file("a.ts") x2  → unsafe (同一文件)                    │
│                                                                │
│ → 这是最智能的方案：同一批工具里，根据具体输入判断安全性        │
└────────────────────────────────────────────────────────────────┘
```

### 伪代码：Claude Code 的并发安全判断

```typescript
// 简化版并发安全判断逻辑

class BashTool {
    isConcurrencySafe(input: { command: string }): boolean {
        const cmd = input.command;
        
        // 有管道、链接、重定向 → 不安全
        if (/[|&;><]/.test(cmd)) return false;
        
        // 已知的纯只读命令 → 安全
        if (/^(ls|cat|head|tail|echo|which|pwd)(\s|$)/.test(cmd)) return true;
        
        // 其他 → 不安全（保守策略）
        return false;
    }
}

class EditTool {
    isConcurrencySafe(input: { file_path: string }): boolean {
        // 检查这一批里有没有编辑同一个文件的
        const sameFileCalls = this.currentBatch.filter(
            c => c.name === "edit_file" && c.input.file_path === input.file_path
        );
        return sameFileCalls.length <= 1;  // 只有自己 → 安全
    }
}

class GrepTool {
    isConcurrencySafe(): boolean {
        return true;  // 搜索永远安全
    }
}

class ReadTool {
    isConcurrencySafe(): boolean {
        return true;  // 读取永远安全
    }
}
```

## 四、关键差异总结

| 维度 | Codex | Pi-Mono | Claude Code |
|------|-------|---------|-------------|
| **文件隔离** | OS 沙箱 | 无（会话分支） | Git worktree |
| **进程隔离** | 独立 Codex 实例 | 无 | AgentTool 子代理 |
| **工具并发** | 串行 | parallel/sequential 二选一 | 智能分批 |
| **冲突检测** | 沙箱阻止 | 无 | isConcurrencySafe |
| **事件流** | tokio channel | 无 | Zustand 状态 |
| **清理策略** | 无（沙箱自动）| 无 | 无变更自动清理 worktree |

## 五、给你的思考题

1. **为什么 Claude Code 的 isConcurrencySafe 是每个工具自己实现，而不是全局的？**
   - 提示：`bash` 的安全性取决于具体命令内容，不能用工具名笼统判断

2. **Git worktree vs Docker 容器，哪个更适合 agent 隔离？**
   - worktree：轻量、共享 .git、方便合并，但只隔离文件
   - Docker：完全隔离（文件+进程+网络），但重量级、不方便合并代码

3. **Pi-Mono 的"全并行"模式有什么风险？**
   - 如果 5 个工具里有 2 个编辑同一个文件 → 竞争条件
   - 如果 bash 有副作用 → 执行顺序可能影响结果
