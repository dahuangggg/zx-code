## 一、流式传输实现

learn-claude-code 的教学中没有深入流式传输。但这是生产系统的核心。

### 为什么需要流式？

```
非流式：用户输入 → [等待 30 秒] → 完整回复出现
流式：  用户输入 → 立即开始逐字出现 → 30 秒后完成

流式不仅是体验问题，还是功能问题：
- 用户可以看到模型的"思考过程"
- 可以提前发现错误方向并中断
- 工具执行的进度可以实时展示
```

### 三种流式方案

```
┌────────────────────────────────────────────────────────────┐
│ Codex: 双传输层                                             │
│                                                            │
│ 优先: WebSocket（全双工，低延迟）                           │
│   → tokio-tungstenite 库                                   │
│   → 支持连接预热（generate=false 先建连接）                │
│   → x-codex-turn-state header 做 sticky 路由              │
│                                                            │
│ 降级: HTTP/SSE（Server-Sent Events）                        │
│   → eventsource-stream 库                                  │
│   → WebSocket 失败时自动切换                                │
│   → 每次 turn 一个 HTTP 请求                               │
│                                                            │
│ 优势:                                                      │
│   - WebSocket 比 SSE 延迟更低（无 HTTP 开销）              │
│   - 预热机制减少首次响应延迟                                │
│   - 自动降级保证可用性                                     │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│ Pi-Mono: 供应商 SDK 原生流                                  │
│                                                            │
│ 每个供应商有自己的流式实现:                                  │
│                                                            │
│ // packages/ai/src/stream.ts                                │
│ async function* stream(context: Context): Stream<Event> {   │
│     const provider = getProvider(context.model);            │
│     yield* provider.stream(context);                        │
│     // 每个 provider 自己处理流式                           │
│     // Anthropic: SSE                                       │
│     // OpenAI: SSE                                          │
│     // Google: gRPC stream                                  │
│ }                                                          │
│                                                            │
│ // 统一事件类型:                                             │
│ type StreamEvent =                                          │
│     | { type: "text_delta", text: string }                  │
│     | { type: "thinking_delta", thinking: string }          │
│     | { type: "toolcall_delta", id: string, name: string }  │
│     | { type: "usage", input: number, output: number }      │
│     | { type: "done" }                                      │
│                                                            │
│ 优势:                                                      │
│   - 支持 15+ 供应商的流式                                   │
│   - 统一的事件接口                                          │
│   - 每个供应商用最优的传输方式                              │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│ Claude Code: Anthropic SDK 流 + React Ink 渲染              │
│                                                            │
│ // 使用 Anthropic 官方 SDK 的流式接口                       │
│ const stream = await claude.messages.stream({               │
│     model: "claude-sonnet-4-20250514",                      │
│     messages: messages,                                     │
│     tools: tools,                                           │
│ });                                                        │
│                                                            │
│ // Async generator 逐事件 yield                             │
│ for await (const event of stream) {                         │
│     yield event;  // React Ink 组件消费并渲染               │
│ }                                                          │
│                                                            │
│ // React Ink 组件实时更新终端:                               │
│ // - 文本逐字出现                                           │
│ // - 工具调用显示进度条                                     │
│ // - 权限弹窗可以在流式过程中出现                           │
│                                                            │
│ 优势:                                                      │
│   - React 的声明式渲染天然适合增量更新                      │
│   - 可以在流式过程中插入 UI 交互                            │
│   - Zustand 状态管理保证 UI 一致性                          │
└────────────────────────────────────────────────────────────┘
```

## 二、多模型支持

```
Codex:      仅 OpenAI (GPT-4o, o3 等)
            + ChatGPT, LMStudio, Ollama 集成
            → 本质上只支持 OpenAI Responses API 格式

Pi-Mono:    15+ 供应商
            Anthropic, OpenAI, Google (Gemini/Vertex), 
            Azure, Bedrock, Mistral, Groq, Cerebras, 
            xAI, OpenRouter, HuggingFace, Kimi, MiniMax
            → 统一的 Model 接口抽象
            → 每个供应商有独立的 adapter

Claude Code: 仅 Claude (Anthropic)
            → 深度集成 Claude 特性
            → 如 extended thinking, tool_use 原生支持
```

### Pi-Mono 的多模型抽象

```typescript
// packages/ai/src/types.ts

interface Model {
    id: string;           // "claude-sonnet-4-20250514"
    name: string;         // "Claude Sonnet 4"
    provider: string;     // "anthropic"
    api: string;          // "anthropic-messages"
    contextWindow: number; // 200000
    cost: {
        input: number;    // 每 1M token 的价格
        output: number;
    };
    reasoning: boolean;   // 是否支持思维链
    inputTypes: string[]; // ["text", "image"]
    outputTypes: string[];
}

// 供应商适配器模式:
// packages/ai/src/providers/
// ├── anthropic.ts      → Messages API
// ├── openai.ts         → Chat Completions API
// ├── google.ts         → Gemini API
// ├── azure-openai.ts   → Azure API
// ├── bedrock.ts        → AWS Bedrock
// ├── vertex.ts         → Google Vertex AI
// └── ... (17+ files)

// 每个适配器实现统一接口:
interface Provider {
    stream(context: Context): AsyncGenerator<StreamEvent>;
    // 把各家 API 的差异封装在内部
}
```

## 三、终端 UI 实现

```
Codex:      ratatui (Rust TUI 框架)
            → codex-rs/tui/
            → 高性能，原生终端渲染

Pi-Mono:    自建 TUI 引擎 (packages/tui)
            → 41KB 的核心引擎
            → 差分渲染（只更新变化的部分）
            → 支持终端图片、IME 输入、kill-ring
            → 自建 Markdown 渲染器

Claude Code: React Ink
            → 用 React 组件模型渲染终端
            → 声明式 UI（状态变 → 自动重渲染）
            → Zustand 状态管理
            → 146+ 组件文件
```

### 为什么 Pi-Mono 要自建 TUI？

```
原因:
1. 现有 TUI 库不够灵活
   → 需要差分渲染（终端不像浏览器有 DOM diffing）
   → 需要精确的光标控制

2. 特殊需求
   → 终端内图片显示（iTerm2/Kitty 协议）
   → IME 输入支持（中文/日文）
   → 自定义 Markdown 渲染

3. 最终控制
   → 不依赖外部库的更新周期
   → 可以精确优化性能
```

## 五、代码智能（LSP 集成）

```
Codex:      无 LSP 集成
Pi-Mono:    无 LSP 集成
Claude Code: 内置 LSP 工具
            → LSP hover（查看类型信息）
            → LSP goto definition（跳转定义）
            → LSP references（查找引用）
```

Claude Code 是唯一集成了 LSP（Language Server Protocol）的 agent。这意味着模型可以：

```typescript
// 模型可以调用 LSP 工具:

// 查看某个符号的类型
LSP.hover({ file: "src/app.ts", line: 42, column: 15 })
// → "const app: Express"

// 跳转到定义
LSP.gotoDefinition({ file: "src/app.ts", line: 42, column: 15 })
// → "node_modules/express/index.d.ts:123"

// 查找所有引用
LSP.references({ file: "src/types.ts", line: 10, symbol: "User" })
// → ["src/api/users.ts:5", "src/models/user.ts:1", ...]
```

## 七、各仓库独有的亮点

### Codex 独有

```
1. 操作系统级沙箱（Seatbelt/Landlock/RestrictedToken）
   → 其他两个完全没有

2. WebSocket 流式 + SSE 降级
   → 双传输层自动切换

3. 企业 Guardian 审批
   → IT 管理员集中控制

4. 延迟网络审批
   → 乐观执行 + 异步审批

5. MCP 服务器模式
   → Codex 可以被其他工具调用
```

### Pi-Mono 独有

```
1. 15+ LLM 供应商统一抽象
   → 最广泛的模型支持

2. 会话树 + /fork + /tree
   → 可视化的探索分支管理

3. Extension API（全面的扩展接口）
   → registerTool/registerCommand/registerShortcut/setWidget/...
   → 15+ 种扩展能力

4. Pi Packages（完整的包管理）
   → npm/git 安装扩展

5. 自建 TUI 引擎
   → 终端图片、IME、差分渲染

6. BashOperations 接口
   → 可替换为 SSH/远程执行
```

### Claude Code 独有

```
1. 7 层权限规则源
   → CLI > 会话 > 本地 > 项目 > 用户 > 企业 > 功能标志

2. StreamingToolExecutor 并发安全分类
   → 按工具输入动态判断是否可并行

3. LSP 集成
   → hover/gotoDefinition/references

4. ScheduleWakeup（自我调度）
   → 模型可以告诉系统 "N 秒后叫醒我"
   → 缓存感知的延迟选择

5. CLAUDE.md 指令链
   → 从根目录到当前目录逐层读取

6. system-reminder 动态提醒
   → 不影响 prompt cache 的每轮更新

7. React Ink 终端 UI
   → 声明式渲染 + Zustand 状态管理

8. AgentTool 类型化子代理
   → Explore/Plan/Review/general-purpose

9. Git worktree 隔离
   → 子代理在独立分支工作

10. Speculative classifier
    → Auto 模式下自动预测用户是否会同意
```

## 八、总结：三个仓库的"一句话精髓"

```
Codex:
  "用 Rust 的性能和 OS 的安全，构建企业级 agent"
  强项: 安全、性能、企业管控
  弱项: 仅支持 OpenAI 模型、扩展能力有限

Pi-Mono:
  "最小内核 + 最大扩展性，让开发者自由定制"
  强项: 多模型、扩展 API、轻量灵活
  弱项: 无沙箱、无多代理、无 MCP

Claude Code:
  "流式精细控制，全方位辅助开发者"
  强项: 权限精细、工具丰富、MCP 生态、规划能力
  弱项: 仅支持 Claude 模型、代码量大（100K+ LoC）
```

### 选择建议

```
如果你重视安全和企业合规    → Codex
如果你想支持多种模型        → Pi-Mono
如果你想要最丰富的功能      → Claude Code
如果你想学习如何构建 Agent  → learn-claude-code + 本系列
```
