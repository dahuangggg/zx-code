## 一、问题引入

learn-claude-code 把多智能体分成四个递进层次：

```
s04 Subagent       → 用完就丢的子进程（"帮我查个东西"）
s15 Agent Teams    → 持久存在的队友（"团队里有前端和后端工程师"）
s16 Team Protocols → 队友间的协作规矩（"先审批再合并"）
s17 Autonomous     → 自己找活干的智能体（"没人指挥也能工作"）
```

这四层在生产系统中的实现差异巨大。

## 二、s04 Subagent — 一次性子代理

### learn-claude-code 的教学

```python
# 核心思想：子代理有空白的 messages[]，只返回摘要
def run_subagent(task: str) -> str:
    child_messages = [
        system("你是一个助手"),
        user(task),
    ]
    
    # 子代理独立运行完整的 agent loop
    while True:
        response = call_llm(child_messages)
        if no_tool_calls(response):
            break
        for tool_call in response.tool_calls:
            result = execute(tool_call)
            child_messages.append(tool_result(result))
    
    # 只返回最终摘要，不返回完整历史
    return response.content  # ← 关键：隔离上下文
```

**核心洞察**：子代理的存在是为了**保护父代理的 context window**。如果把所有工作都在一个 messages[] 里做，context 会爆炸。

### 三个仓库的 Subagent 实现

### 2.1 Codex — agent_jobs（重量级子代理）

```rust
// codex-rs/tools/handlers/agent_jobs.rs

// Codex 的子代理是完整的 Codex 实例
async fn spawn_agent(name: &str, task: &str) -> AgentJob {
    // 1. 创建新的 Codex 实例（独立的 session_loop）
    let child_config = CodexConfig {
        model: parent.model.clone(),
        system_prompt: task_specific_prompt(task),
        tools: parent.tools.clone(),  // 继承父代理的工具
        sandbox: parent.sandbox.clone(),
        // 但不共享 history！
    };
    
    let child = Codex::spawn(child_config).await;
    
    // 2. 提交任务
    child.submit(Op::UserTurn { input: task.into() }).await;
    
    // 3. 监听子代理事件，流式转发给父代理
    tokio::spawn(async move {
        while let Some(event) = child.next_event().await {
            match event {
                Event::Message(text) => {
                    parent_tx.send(Event::AgentMessage { 
                        agent: name, text 
                    });
                }
                Event::TurnCompleted => {
                    parent_tx.send(Event::AgentJobCompleted { 
                        agent: name, 
                        result: child.last_response(),
                    });
                    break;
                }
                _ => {} // 转发其他事件
            }
        }
    });
    
    AgentJob { id: job_id, child }
}
```

**Codex 的关键设计**：
- 子代理是**完整的 Codex 实例**，有自己的 session_loop
- 支持**多个子代理并行**运行（multi_agents_v2）
- 子代理的事件**流式转发**给父代理（不等完成再返回）
- 支持**层级限制**（防止无限嵌套）

### 2.2 Pi-Mono — 无内置子代理


**Pi-Mono 的设计哲学**：
- 不内置子代理，因为"大部分任务一个 agent 就够了"
- 会话树（session tree）是更轻量的替代方案
- 如果真需要，Extension API 完全可以实现

### 2.3 Claude Code — AgentTool（功能最完整）

```typescript
// src/tools/AgentTool
// Claude Code 的子代理是通过 AgentTool 调用的

class AgentTool implements Tool {
    name = "Agent";
    
    inputSchema = z.object({
        description: z.string(),      // 简短描述
        prompt: z.string(),            // 详细任务
        subagent_type: z.string().optional(),  // 专门化代理类型
        run_in_background: z.boolean().optional(),
        model: z.enum(["sonnet", "opus", "haiku"]).optional(),
        isolation: z.enum(["worktree"]).optional(),  // Git 隔离
    });
    
    async execute(input) {
        // 1. 创建子代理配置
        const childConfig = {
            model: input.model || parentModel,
            systemPrompt: buildSubagentPrompt(input),
            tools: selectTools(input.subagent_type),  // 按类型选工具
            messages: [],  // 空白上下文
        };
        
        // 2. 如果需要 worktree 隔离
        if (input.isolation === "worktree") {
            const worktree = await createGitWorktree();
            childConfig.cwd = worktree.path;
        }
        
        // 3. 前台 or 后台
        if (input.run_in_background) {
            // 后台运行，结果通过通知返回
            const session = spawnBackgroundAgent(childConfig);
            return { content: `Agent ${session.id} started in background` };
        } else {
            // 前台运行，等待完成
            const result = await runAgent(childConfig, input.prompt);
            return { content: result.summary };
        }
    }
}
```

**Claude Code 的子代理特色**：
- **类型化子代理**（`subagent_type`）：
  - `Explore` — 代码探索，只有只读工具
  - `Plan` — 架构设计，没有编辑工具
  - `code-reviewer` — 代码审查
  - `general-purpose` — 通用
- **Git worktree 隔离**：子代理在独立的 git 分支工作
- **前台/后台切换**：同一个工具支持两种模式
- **模型选择**：子代理可以用不同的模型（opus/sonnet/haiku）

## 五、伪代码：构建一个多代理系统

假设你要自己实现，综合三个仓库的最佳实践：

```typescript
// 综合最佳实践的多代理架构伪代码

class MultiAgentSystem {
    private coordinator: Agent;
    private workers: Map<string, Worker>;
    private taskQueue: TaskQueue;
    private messageBus: MessageBus;
    
    // --- 从 Claude Code 学的：类型化代理 ---
    spawnWorker(type: AgentType, task: Task): Worker {
        const config = {
            tools: selectToolsByType(type),  // Explore 只有只读工具
            model: selectModelByTask(task),   // 简单任务用 haiku
            systemPrompt: buildPrompt(type),
            isolation: task.needsIsolation ? "worktree" : "shared",
        };
        return new Worker(config);
    }
    
    // --- 从 Codex 学的：事件驱动通知 ---
    async runWorker(worker: Worker, task: Task) {
        worker.on("event", (event) => {
            // 实时转发事件给协调员
            this.messageBus.send(worker.id, "coordinator", event);
        });
        
        worker.on("approval_needed", async (request) => {
            // 从 Codex 学的：审批流
            const decision = await this.coordinator.review(request);
            worker.submitApproval(decision);
        });
        
        await worker.run(task.prompt);
    }
    
    // --- 从 learn-claude-code 学的：自主循环 ---
    async coordinatorLoop() {
        while (true) {
            // 1. 检查信箱
            const messages = this.messageBus.read("coordinator");
            for (const msg of messages) {
                await this.coordinator.handle(msg);
            }
            
            // 2. 检查任务队列
            const ready = this.taskQueue.findReady();
            for (const task of ready) {
                const type = classifyTask(task);  // 分类任务
                const worker = this.spawnWorker(type, task);
                this.runWorker(worker, task);  // 不 await，后台运行
            }
            
            // 3. 检查超时
            for (const [id, worker] of this.workers) {
                if (worker.idleTime > 60_000) {
                    worker.shutdown();
                    this.workers.delete(id);
                }
            }
            
            // 4. 所有任务完成？
            if (this.taskQueue.allCompleted()) break;
            
            await sleep(5000);
        }
    }
}
```

## 六、关键差异总结

| 维度 | Codex | Pi-Mono | Claude Code |
|------|-------|---------|-------------|
| **子代理** | 完整 Codex 实例 | 无内置（可扩展）| AgentTool 灵活配置 |
| **持久队友** | 无 | 无 | Teammates + Swarm |
| **代理类型化** | 无（通用） | 无 | Explore/Plan/Review 等 |
| **模型选择** | 固定 | 15+ 供应商可选 | opus/sonnet/haiku 分级 |
| **隔离方式** | 沙箱隔离 | 无 | Git worktree 隔离 |
| **通信方式** | 事件通道 | 无 | 消息总线 + React 状态 |
| **自主性** | 无 | 无 | ScheduleWakeup 自我调度 |
| **协调机制** | Guardian 审批 | 无 | Coordinator + 自动化检查 |

## 七、给你的思考题

1. **为什么 Claude Code 要让子代理选择不同的模型？**
   - 提示：搜索只需要 haiku（便宜快速），代码审查需要 opus（聪明但贵）

2. **Git worktree 隔离 vs OS 沙箱隔离，分别适合什么场景？**
   - worktree：防止代码冲突（多个 agent 改同一个文件）
   - 沙箱：防止安全逃逸（agent 不能访问不该访问的东西）

3. **Pi-Mono "不做多代理" 的选择是否明智？**
   - 提示：多代理协调的复杂度是指数级的。大部分 coding 任务，一个 agent 够了
