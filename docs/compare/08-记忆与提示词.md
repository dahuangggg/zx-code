# 08 — 记忆系统与提示词管线

> 对应 learn-claude-code: **s09 Memory System** + **s10 System Prompt**

---

## 一、问题引入

Agent 需要两种"知识"：

1. **短期知识** — 当前对话的上下文（messages）
2. **长期知识** — 跨会话的持久记忆（用户偏好、项目约定、历史决策）

learn-claude-code 把这两个问题分成了两章：
- s09: Memory — 四类持久记忆
- s10: System Prompt — 如何把所有知识组装成一个提示词

### 三个仓库的记忆实现

### 2.1 Codex — memories 模块（压缩时提取）

```rust
// codex-rs/memories/

// Codex 的记忆系统特色：在压缩时自动提取

// 记忆生命周期:
// 1. 对话进行中...
// 2. context 超限，触发 compact
// 3. compact 过程中，从旧历史里提取"长期记忆"
// 4. 记忆写入 memories 存储
// 5. 下次 compact 后，memories 注入新 context

// 伪代码:
async fn compact_with_memory_extraction(history: &[Message]) -> CompactResult {
    // 1. 做常规压缩
    let summary = summarize(history).await;
    
    // 2. 同时提取长期记忆
    let memories = extract_memories(history).await;
    // extract_memories 用 LLM 识别值得长期保存的事实:
    // - 用户偏好
    // - 项目约定
    // - 重要决策
    
    // 3. 合并到已有记忆
    memory_store.merge(memories).await;
    // → 去重、更新过期的、删除矛盾的
    
    CompactResult { summary, new_memories: memories }
}

// 记忆在下次构建 context 时注入:
fn build_context(history: &[Message], memories: &[Memory]) -> Vec<Message> {
    let mut context = vec![system_prompt()];
    
    // 注入记忆（在 system prompt 后面）
    if !memories.is_empty() {
        context.push(system(format_memories(memories)));
    }
    
    context.extend(history);
    context
}
```

**Codex 的关键设计**：
- 记忆提取是**被动的**（压缩时自动触发，不需要模型主动请求）
- 记忆是**合并的**（新记忆和旧记忆自动合并）
- 存储在 SQLite 中（和会话历史同一个数据库）

## 三、s10 System Prompt — 提示词管线

### learn-claude-code 的教学

```python
# 不要把 system prompt 当字符串拼接，要当管线来组装

def build_system_prompt(state) -> str:
    sections = []
    
    # 1. 核心身份（稳定，很少变）
    sections.append(CORE_IDENTITY)
    
    # 2. 工具目录（会话开始时确定）
    sections.append(format_tools(state.tools))
    
    # 3. 技能名称（便宜，始终包含）
    sections.append(format_skill_names(state.skills))
    
    # 4. 记忆（跨会话持久）
    sections.append(format_memories(state.memories))
    
    # 5. CLAUDE.md 指令链（项目级）
    sections.append(load_claude_md(state.cwd))
    
    # 6. 动态运行时上下文（每次变化）
    sections.append(format_runtime_context(state))
    
    return "\n\n".join(sections)

# 关键洞察：
# 稳定内容 → 放前面（利用 API 缓存）
# 动态内容 → 放后面（不影响缓存命中率）
```

## 四、CLAUDE.md / AGENTS.md — 项目级指令

这是三个仓库都支持的机制，但实现细节不同：

```
文件名约定:
  Codex:       AGENTS.md（符合 OpenAI 约定）
  Pi-Mono:     AGENTS.md 和 CLAUDE.md 都支持
  Claude Code:  CLAUDE.md（Anthropic 约定）

加载方式:
  Codex:       读取项目根目录的 AGENTS.md → 注入 developer instructions
  Pi-Mono:     读取项目根目录的 AGENTS.md 或 CLAUDE.md → 注入 system prompt
  Claude Code:  从项目根到当前 cwd 的每层 CLAUDE.md 都读取，形成"指令链"
                /project/CLAUDE.md → 基础规则
                /project/src/CLAUDE.md → src 目录特有规则
                /project/src/api/CLAUDE.md → api 目录特有规则
                → 越具体的位置，优先级越高

用途:
  - 项目约定（代码风格、测试策略）
  - 禁止事项（不要改某些文件）
  - 偏好设置（用什么包管理器）
  - 架构说明（模块关系）
```

## 五、关键差异总结

| 维度 | Codex | Pi-Mono | Claude Code |
|------|-------|---------|-------------|
| **记忆系统** | 压缩时自动提取 | 无内置（文件替代）| 四类记忆 + MEMORY.md |
| **记忆存储** | SQLite | 无 | Markdown 文件 |
| **记忆触发** | 被动（压缩时）| 无 | 主动（模型写文件）|
| **项目指令** | AGENTS.md (单层) | AGENTS.md/CLAUDE.md | CLAUDE.md (多层链) |
| **提示词缓存** | 不明确 | 不明确 | 明确优化（稳定前/动态后）|
| **系统提示可替换** | 部分（developer instructions）| 完全（SYSTEM.md）| 不可替换（核心固定）|
| **动态提醒** | 无 | 无 | system-reminder 标签 |

## 六、给你的思考题

1. **为什么 Claude Code 的记忆是"模型主动写文件"而不是"系统自动提取"？**
   - 提示：Claude Code 信任模型的判断——模型知道什么值得记住
   - Codex 的自动提取可能提取到不重要的信息

2. **CLAUDE.md 的"指令链"设计有什么好处？**
   - 提示：想想 monorepo——前端和后端可能有不同的规则
   - `/project/CLAUDE.md` → 全局规则
   - `/project/frontend/CLAUDE.md` → 前端特有规则

3. **为什么 Claude Code 把动态内容放在 system-reminder 标签里而不是 system prompt？**
   - 提示：Anthropic API 的 prompt cache 有 5 分钟 TTL
   - system prompt 变了 → 缓存失效 → 重新计算 → 更慢更贵
   - system-reminder 在消息流中 → 不影响 system prompt 缓存
