# 05 — 上下文管理

> 对应 learn-claude-code: **s06 Context Compact**

---

## 一、问题引入

每个 LLM 都有 context window 限制。当对话越来越长，你必须压缩，否则：
- 要么 API 报错（token 超限）
- 要么成本爆炸（按 token 计费）
- 要么质量下降（模型注意力被稀释）

learn-claude-code 教了**四个压缩杠杆**：

```
杠杆 0: Persisted Output — 大结果写磁盘，context 里只放摘要
杠杆 1: Micro-compact   — 旧的工具结果替换为占位符
杠杆 2: Auto-compact    — 整个对话历史压缩为摘要
杠杆 3: Manual compact  — 模型主动请求压缩
```

---

## 二、杠杆 0：大输出持久化

### 问题场景

```
用户: "读取这个 10000 行的日志文件"
Agent: [调用 read_file → 返回 10000 行文本]
模型: "我看到日志里第 5432 行有个错误..."
```

如果把 10000 行全放进 context，一次对话就用掉大量 token。

### 三个仓库的处理

## 三、杠杆 2：自动压缩（Auto-compact）

这是最核心的机制——当 context 接近 token 限制时，自动压缩历史。

### learn-claude-code 的教学

```python
def auto_compact(messages: list) -> list:
    """当 token 数超过阈值时压缩"""
    if count_tokens(messages) < TOKEN_THRESHOLD:
        return messages  # 还没到需要压缩的时候
    
    # 把历史消息发给 LLM 做摘要
    summary = call_llm([
        system("把以下对话压缩为简短摘要，保留关键信息"),
        user(format_messages(messages[:-KEEP_RECENT])),
    ])
    
    # 替换旧消息为摘要
    return [
        system_message(f"[对话摘要]\n{summary}"),
        *messages[-KEEP_RECENT:],  # 保留最近 N 条
    ]
```

### 三个仓库的压缩策略

```typescript
// ========== Codex 的压缩 ==========
// codex-rs/compact/ — 两种模式

// 模式 1: 本地快速压缩
async function compactLocal(history: Message[]): Message[] {
    // 用当前模型做摘要（inline compact）
    const summary = await model.generate({
        prompt: "Summarize this conversation for context continuation",
        messages: history.slice(0, -KEEP_RECENT),
    });
    return [summaryMessage(summary), ...history.slice(-KEEP_RECENT)];
}

// 模式 2: 远端压缩服务
async function compactRemote(history: Message[]): Message[] {
    // 发送到专门的压缩服务（可能用更便宜的模型）
    const response = await fetch(COMPACT_SERVICE_URL, {
        body: JSON.stringify({ messages: history }),
    });
    return response.json().compacted;
}

// 同时会做"记忆提取"
```

```typescript
// ========== Pi-Mono 的压缩 ==========
// packages/coding-agent/src/core/compaction/

// Pi-Mono 的压缩是纯函数式的
function compactMessages(
    messages: AgentMessage[],
    options: CompactOptions
): CompactResult {
    // 1. 分割：保留最近的，压缩旧的
    const recent = messages.slice(-options.keepRecent);
    const old = messages.slice(0, -options.keepRecent);
    
    // 2. 提取文件操作元数据
    const fileOps = extractFileOperations(old);
    // → 哪些文件被读了、改了——这些信息不能丢
    
    // 3. 生成摘要（用 LLM）
    const summary = await summarize(old);
    
    // 4. 构建压缩摘要消息
    return {
        messages: [
            compactionSummaryMessage(summary, fileOps),
            ...recent,
        ],
        details: { fileOps, removedCount: old.length },
    };
}

// Pi-Mono 独有：分支摘要
// 当用户从一个分支切换到另一个分支时
// 旧分支的内容被压缩为 BranchSummaryMessage
function summarizeBranch(branchMessages: AgentMessage[]): BranchSummaryMessage {
    return {
        type: "branchSummary",
        summary: await summarize(branchMessages),
        branchId: branch.id,
    };
}
```

```typescript
// ========== Claude Code 的压缩 ==========
// 多触发点 + token 预算

// 触发点 1：context 接近 token 限制
// 触发点 2：用户手动 /compact 命令
// 触发点 3：模型请求（通过特殊 tool call）

async function compact(messages: Message[], state: AppState): Message[] {
    // 1. 计算 token 预算
    const budget = state.model.contextWindow * 0.8;  // 留 20% 余量
    
    if (countTokens(messages) < budget) return messages;
    
    // 2. 选择压缩范围
    //    保留：system prompt + 最近 N 条 + 当前工具调用
    //    压缩：中间的历史
    const keepHead = messages.filter(m => m.role === "system");
    const keepTail = messages.slice(-RECENT_COUNT);
    const toCompress = messages.slice(keepHead.length, -RECENT_COUNT);
    
    // 3. LLM 摘要
    const summary = await claude.messages.create({
        model: "claude-sonnet",  // 用便宜的模型做压缩
        system: "Compress this conversation. Preserve: file paths, decisions, errors, TODO items.",
        messages: [{ role: "user", content: formatForCompression(toCompress) }],
    });
    
    // 4. 构建压缩后的消息列表
    return [
        ...keepHead,
        { role: "user", content: `[Previous conversation summary]\n${summary}` },
        ...keepTail,
    ];
}
```

## 四、压缩策略的关键差异

```
                     ┌─────────────────────────────────────┐
                     │      压缩时保留什么？                 │
                     ├─────────────────────────────────────┤
                     │                                     │
  Codex              │  摘要 + 记忆提取                     │
  "压缩的同时提取    │  → 重要事实存入 memories 系统         │
   长期记忆"         │  → 下次即使摘要也丢了，memories 还在  │
                     │                                     │
  Pi-Mono            │  摘要 + 文件操作元数据                │
  "不能忘记改了      │  → 哪些文件被读/写了                  │
   哪些文件"         │  → 防止 compact 后重复读文件          │
                     │                                     │
  Claude Code        │  摘要 + system prompt + 近期对话      │
  "保留最重要的      │  → 决策记录、文件路径、错误信息       │
   上下文"           │  → TODO 项不能丢                      │
                     └─────────────────────────────────────┘
```

### 伪代码：压缩后信息恢复的对比

```python
# 压缩后，模型怎么"记住"之前做了什么？

# ====== Codex ======
# 1. 摘要在 messages 里
# 2. memories 提供长期记忆
# 3. SQLite 可以查历史
context = [
    system_prompt,
    summary_message,           # 压缩摘要
    *memories,                 # 从 memories 系统加载的长期记忆
    *recent_messages,          # 最近的消息
]

# ====== Pi-Mono ======
# 1. CompactionSummaryMessage 包含摘要 + 文件操作记录
# 2. 会话树可以回溯
context = [
    system_prompt,
    compaction_summary_msg,    # 包含 summary + fileOps
    *recent_messages,
]
# fileOps 示例:
# { read: ["src/main.ts", "package.json"], modified: ["src/app.ts"] }

# ====== Claude Code ======
# 1. 摘要在 messages 里
# 2. Tasks 系统提供任务状态（文件持久化，不怕压缩）
# 3. Memory 系统提供长期记忆
context = [
    system_prompt_with_memory,  # system prompt 里包含 MEMORY.md
    summary_message,
    *task_status,              # TaskList 可以恢复任务上下文
    *recent_messages,
]
```

## 五、关键差异总结

| 维度 | Codex | Pi-Mono | Claude Code |
|------|-------|---------|-------------|
| **截断** | 硬限制 + 截断提示 | TruncationResult 统一接口 | 多层（工具/执行器/系统）|
| **压缩触发** | token 超限 | token 超限 | token 超限 + 手动 + 模型请求 |
| **压缩模型** | 本地或远端服务 | 当前模型 | 可用便宜模型 |
| **保留策略** | 摘要 + 记忆提取 | 摘要 + 文件操作 | 摘要 + 任务状态 |
| **恢复机制** | memories 系统 | 会话树回溯 | Tasks + Memory |
| **分支压缩** | 无 | 有（BranchSummary）| 无（worktree 物理隔离）|

## 六、给你的思考题

1. **为什么 Pi-Mono 在压缩时特别保留"文件操作记录"？**
   - 提示：如果 agent 忘了自己已经改过 app.ts，它可能会重新读取旧版本或做冲突的修改

2. **Codex 的"压缩时提取记忆"是什么意思？和直接做摘要有什么区别？**
   - 摘要 = 这次对话的缩写（临时的）
   - 记忆 = 跨会话持久的事实（"这个项目用 pnpm"）
   - 压缩是提取记忆的最佳时机，因为你正在审阅整个历史

3. **为什么 Claude Code 用便宜模型做压缩？**
   - 压缩是"理解"任务，不是"创造"任务
   - sonnet 做摘要已经够好了，没必要用 opus
