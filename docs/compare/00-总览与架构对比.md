# 总览：三大 Coding Agent 架构深度对比

> 本系列以 [learn-claude-code](../learn-claude-code) 的 19 章课程为骨架，
> 逐一拆解 **Codex**（OpenAI）、**Pi-Mono**（pi.dev）、**Claude Code**（Anthropic）
> 三个生产级 Agent 是怎么把同一套核心机制做到可上线的。

---

## 一、三个仓库一句话画像

| 维度 | Codex | Pi-Mono | Claude Code |
|------|-------|---------|-------------|
| 作者 | OpenAI | pi.dev (个人) | Anthropic |
| 语言 | **Rust** 核心 + TS 薄壳/SDK | **TypeScript** 全栈 | **TypeScript + React Ink** |
| 模型 | GPT-4o / o3 (Responses API) | 15+ 家 LLM（Anthropic/OpenAI/Gemini/Bedrock…） | Claude (Messages API) |
| 架构风格 | 单体巨 crate（89 crate 分层）| 分层 monorepo（7 package）| 单 src/ 大目录（55+ 子系统）|
| 核心理念 | **安全沙箱 + 审批流** | **最小核心 + 扩展万物** | **流式 + 精细权限 + MCP 生态** |
| 代码量级 | ~70k+ LoC Rust | ~30k LoC TS | ~100k+ LoC TS |
| 扩展方式 | 技能目录 + MCP | Extension API + Pi Package | Hooks + Skills + Plugins + MCP |

## 三、总体架构图对比

### 3.1 Codex — "层层审批的堡垒"

```
┌──────────────────────────────────────────────────┐
│  入口层: CLI (TS) / app-server (JSON-RPC) / SDK  │
└───────────────┬──────────────────────────────────┘
                ↓
┌──────────────────────────────────────────────────┐
│  核心层: codex.rs                                │
│  ┌─────────────────────────────────────────────┐ │
│  │ submission_loop() ← Op enum (22 种操作)     │ │
│  │   ↓                                         │ │
│  │ handler dispatch → turn_loop()              │ │
│  │   ↓                                         │ │
│  │ ModelClient::stream() (WebSocket/SSE 双通道)│ │
│  │   ↓                                         │ │
│  │ ToolOrchestrator::run()                     │ │
│  │   ├── approval check (execpolicy 规则引擎)  │ │
│  │   ├── sandbox select (Seatbelt/Landlock)    │ │
│  │   └── tool handler (30+ handlers)           │ │
│  └─────────────────────────────────────────────┘ │
│  持久层: SQLite (会话历史) + rollout 文件        │
└──────────────────────────────────────────────────┘
```

**关键词: Submission-Event 模型、平台级沙箱、规则引擎审批**

### 3.2 Pi-Mono — "最小内核 + 扩展一切"

```
┌───────────────────────────────────────────────────┐
│  packages/coding-agent (CLI 入口)                 │
│  ┌──────────────────────────────────────────────┐ │
│  │ InteractiveMode (4749 行 TUI)                │ │
│  │   ↓                                          │ │
│  │ AgentSession → AgentSessionRuntime           │ │
│  │   ↓                                          │ │
│  │ packages/agent: Agent + AgentLoop            │ │
│  │   ├── runAgentLoop() — 异步生成器            │ │
│  │   ├── streamAssistantResponse()              │ │
│  │   └── executeToolCalls() — 并行/串行模式      │ │
│  │   ↓                                          │ │
│  │ packages/ai: stream() — 15+ 供应商统一抽象   │ │
│  └──────────────────────────────────────────────┘ │
│  扩展层: Extension API (工具/命令/快捷键/UI)      │
│  会话层: JSONL 树形会话 + 分支/fork               │
└───────────────────────────────────────────────────┘
```

**关键词: 多供应商统一抽象、Extension API 万物可扩展、JSONL 会话树**

### 3.3 Claude Code — "流式精细控制"

```
┌───────────────────────────────────────────────────┐
│  main.tsx (React Ink 终端 UI)                     │
│  ┌──────────────────────────────────────────────┐ │
│  │ QueryEngine.submitMessage()                  │ │
│  │   ↓                                          │ │
│  │ query() — async generator 核心循环            │ │
│  │   ├── 构建 system prompt 管线                │ │
│  │   ├── Claude API streaming                   │ │
│  │   ├── StreamingToolExecutor                  │ │
│  │   │   ├── 并发安全分类 (读/写分批)           │ │
│  │   │   ├── 权限检查 (7 层规则源)              │ │
│  │   │   └── 45+ 工具执行                       │ │
│  │   └── yield Message/StreamEvent              │ │
│  └──────────────────────────────────────────────┘ │
│  扩展: Hooks + Skills + Plugins + MCP             │
│  状态: Zustand store (30+ 字段)                   │
└───────────────────────────────────────────────────┘
```

**关键词: Async Generator 流式、并发安全分批、7 层权限源、React Ink UI**

## 四、设计哲学对比

### 安全 vs 灵活 vs 精细

```
           安全优先
              ↑
              │
        ┌─────┐
        │Codex│  ← 操作系统级沙箱 + 规则引擎
        └─────┘
              │
              │         ┌───────────┐
              │         │Claude Code│ ← 7 层权限 + 分类器
              │         └───────────┘
              │
 灵活───────────────────────────────── 精细
              │
        ┌───────┐
        │Pi-Mono│ ← 无内置沙箱，靠扩展和容器
        └───────┘
```

| 设计取舍 | Codex | Pi-Mono | Claude Code |
|---------|-------|---------|-------------|
| "不信任模型" | 强：OS 沙箱隔离 | 弱：信任用户环境 | 中：权限 + 钩子 |
| "不信任用户" | 强：企业 Guardian | 弱：完全信任 | 中：规则分层 |
| "需要扩展" | 中：Skills + MCP | 强：Extension API 全覆盖 | 强：4 种扩展机制 |
| "多模型支持" | 弱：仅 OpenAI | 强：15+ 供应商 | 弱：仅 Claude |

## 五、阅读建议

**如果你是初学者**：
1. 先读完 [learn-claude-code](../learn-claude-code) 的 s01-s06，建立核心概念
2. 然后按本系列 01→10 的顺序阅读，每章对照三个仓库的实现

**如果你想快速了解某个功能**：
- 直接跳到对应的 `.md` 文件，每个文件都是独立的

**如果你想自己实现一个 Agent**：
- 先读 01（循环）→ 02（工具）→ 06（权限）→ 05（上下文）
- 这四个就够你写出一个能用的 Agent 了

## 六、名词约定

| 术语 | 本系列中的含义 |
|------|--------------|
| Agent Loop | 发送消息 → 模型回复 → 执行工具 → 回写结果 → 循环 |
| Tool | 模型可以调用的函数（读文件、跑命令等）|
| Turn | 一轮对话（用户输入 → 模型完成所有工具调用 → 最终回复）|
| Context | 当前送给模型的全部信息（系统提示 + 历史消息 + 工具结果）|
| Compact | 压缩上下文以避免超出 token 限制 |
| Submission | Codex 中的概念：一个待处理的操作请求 |
| Extension | Pi-Mono 中的概念：可注册工具/命令/UI 的插件 |
| Hook | 在工具执行前后触发的外部脚本 |
| MCP | Model Context Protocol，模型与外部工具的标准协议 |
