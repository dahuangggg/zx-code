# 06 — 权限与沙箱

> 对应 learn-claude-code: **s07 Permission System**

---

## 一、问题引入

模型可以调用工具。工具可以删文件、跑命令、发网络请求。**如果模型出错了呢？**

learn-claude-code 教了一个四阶段管线：

```
工具调用请求
  ↓
阶段 1: Deny 规则（黑名单，不可绕过）
  ↓ 通过
阶段 2: Mode 检查（plan/auto/default）
  ↓ 通过
阶段 3: Allow 规则（白名单）
  ↓ 不在白名单
阶段 4: Ask 用户
  ↓ 用户同意/拒绝
执行 or 阻止
```

加上断路器（circuit breaker）：连续 3 次被拒绝就停止尝试。

## 二、三大仓库的权限体系

### 2.1 Codex — 操作系统级沙箱 + 规则引擎

Codex 的权限系统是**最安全的**，因为它在**操作系统层面**隔离：

```
┌─────────────────────────────────────────────────────────────┐
│ Codex 权限架构                                               │
│                                                             │
│ 层 1: ExecPolicy 规则引擎 (execpolicy.toml)                 │
│   ┌─────────────────────────────────────────────┐           │
│   │ [[rules]]                                   │           │
│   │ cmd = ["npm", "install"]                    │           │
│   │ decision = "allow"                          │           │
│   │                                             │           │
│   │ [[rules]]                                   │           │
│   │ cmd = ["rm", "-rf"]                         │           │
│   │ decision = "deny"                           │           │
│   │ reason = "Destructive command"              │           │
│   └─────────────────────────────────────────────┘           │
│                                                             │
│ 层 2: Approval Policy (配置文件)                             │
│   approval_policy = "on-request"                            │
│   # always: 每次都审批                                      │
│   # on-request: 工具请求时审批                              │
│   # on-failure: 失败时审批（乐观执行）                      │
│   # never: 永不审批（全自动）                               │
│                                                             │
│ 层 3: OS 沙箱 (sandbox)                                     │
│   ┌─────────────────────────────────────────────┐           │
│   │ macOS: sandbox-exec (Seatbelt SBPL)         │           │
│   │   (deny default)                            │           │
│   │   (allow file-read* (subpath "/project"))   │           │
│   │   (allow file-write* (subpath "/project"))  │           │
│   │   (deny file-write* (subpath ".git"))       │           │
│   │   (deny network*)  ; 或按域名白名单         │           │
│   ├─────────────────────────────────────────────┤           │
│   │ Linux: bubblewrap + Landlock                │           │
│   │   bwrap --ro-bind / /                       │           │
│   │        --bind /project /project             │           │
│   │        --unshare-net                        │           │
│   ├─────────────────────────────────────────────┤           │
│   │ Windows: Restricted Token                   │           │
│   │   CreateRestrictedToken(...)                │           │
│   │   Private Desktop isolation                 │           │
│   └─────────────────────────────────────────────┘           │
│                                                             │
│ 层 4: Guardian (企业级)                                      │
│   → 集中式审批服务                                          │
│   → IT 管理员定义全局策略                                   │
│   → 审计日志                                                │
│                                                             │
│ 层 5: 网络策略                                               │
│   NetworkSandboxPolicy {                                    │
│     allowed_domains: ["npmjs.org", "github.com"]            │
│   }                                                         │
│   → 即使工具需要网络，也只能访问白名单域名                  │
│   → 支持"延迟审批"：先乐观执行，异步获取批准                │
└─────────────────────────────────────────────────────────────┘
```

**伪代码**：

```rust
// Codex 的完整审批流程
async fn approve_and_execute(tool_call: ToolCall) -> Result<Output> {
    // 1. ExecPolicy 规则匹配
    let policy_decision = exec_policy.check(&tool_call.command);
    match policy_decision {
        Decision::Allow => {},  // 继续
        Decision::Deny(reason) => return Err(Denied(reason)),
        Decision::Unmatched => {
            // 2. 检查审批策略
            match config.approval_policy {
                ApprovalPolicy::Never => {},  // 直接执行
                ApprovalPolicy::Always | ApprovalPolicy::OnRequest => {
                    // 3. 请求用户审批
                    let approval = request_approval(&tool_call).await;
                    if !approval.granted { return Err(Denied("User rejected")); }
                }
                ApprovalPolicy::OnFailure => {},  // 乐观执行，失败再审批
            }
        }
    }
    
    // 4. 选择沙箱
    let sandbox = sandbox_manager.select_for(&tool_call);
    
    // 5. 在沙箱内执行
    let result = sandbox.execute(&tool_call).await;
    
    // 6. 如果沙箱拒绝，尝试升级
    if result.is_sandbox_denied() {
        let relaxed = sandbox_manager.escalate(sandbox);
        // 重新请求审批（因为要放宽限制）
        let approval = request_approval_for_escalation(&tool_call).await;
        if approval.granted {
            return relaxed.execute(&tool_call).await;
        }
    }
    
    result
}
```

### 2.3 Claude Code — 7 层规则源 + 分类器

```
┌─────────────────────────────────────────────────────────────┐
│ Claude Code 权限架构                                         │
│                                                             │
│ 决策流程 (canUseTool):                                       │
│                                                             │
│ tool_call 到达                                               │
│   ↓                                                         │
│ 1. always-allow 规则检查 → 匹配则直接允许                   │
│ 2. always-deny 规则检查  → 匹配则直接拒绝                   │
│ 3. always-ask 规则检查   → 匹配则必须问用户                 │
│ 4. 模式检查:                                                │
│    ├── plan 模式 → 只允许只读工具                           │
│    ├── auto 模式 → 分类器自动判断                           │
│    └── default 模式 → 问用户                                │
│   ↓                                                         │
│ 如果需要问用户:                                              │
│   ├── Coordinator worker → 自动化检查                       │
│   ├── Swarm worker → 统一权限处理                           │
│   └── Interactive → 终端弹出对话框                          │
│       用户可以选择:                                          │
│       - "Allow once" → 这次允许                              │
│       - "Allow always" → 记住规则                            │
│       - "Deny" → 这次拒绝                                   │
│                                                             │
│ 7 层规则源（优先级从高到低）:                                │
│   1. CLI 标志                                                │
│   2. 会话临时授权                                            │
│   3. 本地设置 (.claude/settings.local.json)                  │
│   4. 项目设置 (.claude/settings.json)                        │
│   5. 用户设置 (~/.claude/settings.json)                      │
│   6. 企业策略 (MDM 下发)                                     │
│   7. 功能标志                                                │
│                                                             │
│ 规则类型:                                                    │
│   - 路径模式: { "path": "src/**", "allow": true }           │
│   - 工具名: { "tool": "bash", "ask": true }                 │
│   - 命令模式: { "command": "rm *", "deny": true }           │
│                                                             │
│ 额外功能:                                                    │
│   - Auto 模式的 speculative classifier:                      │
│     → 用分类器预测用户是否会同意                             │
│     → 高置信度自动放行                                      │
│   - Hook 系统:                                               │
│     → permission_request 钩子可以自定义审批逻辑             │
└─────────────────────────────────────────────────────────────┘
```

**伪代码**：

```typescript
// Claude Code 的权限检查流程
async function canUseTool(
    tool: Tool,
    input: any,
    context: PermissionContext
): Promise<PermissionDecision> {
    // 从 7 个来源收集规则
    const rules = mergeRules([
        context.cliFlags,           // 最高优先
        context.sessionGrants,
        context.localSettings,
        context.projectSettings,
        context.userSettings,
        context.enterprisePolicy,
        context.featureFlags,       // 最低优先
    ]);
    
    // 按优先级检查规则
    for (const rule of rules) {
        if (rule.matches(tool.name, input)) {
            if (rule.type === "allow") return { behavior: "allow" };
            if (rule.type === "deny") return { behavior: "deny" };
            if (rule.type === "ask") break;  // 进入交互流程
        }
    }
    
    // 没有匹配的规则，检查模式
    switch (context.mode) {
        case "plan":
            // plan 模式只允许只读工具
            return tool.isReadOnly ? { behavior: "allow" } : { behavior: "deny" };
        
        case "auto":
            // 用分类器自动判断
            const prediction = await classifier.predict(tool, input);
            if (prediction.confidence > 0.9) {
                return { behavior: prediction.decision };
            }
            // 低置信度仍然问用户
            break;
        
        case "default":
            break;  // 问用户
    }
    
    // 需要问用户
    return { behavior: "ask" };
}
```

## 三、沙箱实现细节对比

### 为什么只有 Codex 做了 OS 级沙箱？

```
原因分析:

Codex (OpenAI):
  - 面向企业客户，安全是卖点
  - Rust 语言适合做底层系统编程
  - 团队有操作系统安全背景
  → 所以做了 Seatbelt + Landlock + RestrictedToken

Pi-Mono (个人项目):
  - 面向开发者，信任用户
  - TypeScript 不方便做 OS 级操作
  - 设计哲学："在容器里跑就行了"
  → 所以不做沙箱，提供接口让你自己封装

Claude Code (Anthropic):
  - 面向开发者，但需要安全保障
  - TypeScript 做 OS 沙箱太重了
  - 选择了"权限规则 + 用户确认"的软防护
  → 多层规则源 + 分类器 + 钩子
```

### 沙箱的层次对比

```
             硬件
              ↑
     OS 内核 (Seatbelt/Landlock)  ← Codex 在这里
              ↑
     容器 (Docker/bubblewrap)     ← Pi-Mono 推荐在这里
              ↑
     进程隔离                      ← 子代理的 messages 隔离
              ↑
     权限规则 (软件层)             ← Claude Code 在这里
              ↑
     用户确认 (人在回路)           ← 所有三个都有
```

**越往下越安全，但也越难实现、越不灵活。**

## 四、关键差异总结

| 维度 | Codex | Pi-Mono | Claude Code |
|------|-------|---------|-------------|
| **防护层次** | OS 内核级 | 容器/接口级 | 软件规则级 |
| **规则系统** | execpolicy.toml | 无 | 7 层规则源 |
| **审批模式** | always/on-request/on-failure/never | beforeToolCall 钩子 | plan/auto/default |
| **自动审批** | 无 | 无 | speculative classifier |
| **网络控制** | 域名白名单 | 无 | 无内置 |
| **企业管控** | Guardian | 无 | MDM 策略 |
| **沙箱逃逸风险** | 极低（OS 强制）| 取决于容器配置 | 中等（软件可绕过）|
| **开发体验** | 严格（可能误拦）| 宽松（全靠自觉）| 平衡（交互确认）|

## 五、给你的思考题

1. **Codex 的"延迟网络审批"是什么意思？有什么好处？**
   - 提示：先发网络请求，同时异步问用户"你同意吗？"
   - 如果用户同意，请求已经完成了；如果拒绝，回滚结果
   - → 减少用户等待时间

2. **Claude Code 的 7 层规则源为什么设计成这样？**
   - 提示：想想不同场景——
     - 你在公司电脑上，IT 有全局策略（企业策略）
     - 项目有自己的安全要求（项目设置）
     - 你个人有偏好（用户设置）
     - 这次对话你临时允许了某个操作（会话授权）

3. **如果你只能选一种权限方案，你选哪个？**
   - OS 沙箱最安全，但实现成本高
   - 规则系统最灵活，但可能被绕过
   - 用户确认最简单，但会打断工作流
