## 二、s03 TodoWrite — 会话内规划

### learn-claude-code 的教学

```python
class TodoManager:
    def __init__(self):
        self.items = []  # [{id, text, status}]
    
    def update(self, items):
        """模型调用 todo_write 时更新清单"""
        self.items = items
    
    def render(self) -> str:
        """渲染成文本注入到 system prompt"""
        lines = ["## Current Plan"]
        for item in self.items:
            mark = "✓" if item.status == "done" else "○"
            lines.append(f"  {mark} {item.text}")
        return "\n".join(lines)
    
    # 关键规则：
    # 1. 只能有一个 in_progress 项
    # 2. 每 3 轮没更新就催促（nag）
```

## 三、任务系统详细对比

### 3.1 Claude Code 的 Task 系统伪代码

```typescript
// Claude Code 的完整任务生命周期

// 1. 创建任务
const task = await TaskCreate({
    description: "重构支付模块",
    blockedBy: ["task-auth"],  // 等认证模块完成
});
// → 写入 .tasks/task-pay.json
// → 状态: pending（被 task-auth 阻塞）

// 2. 上游完成时自动解锁
await TaskUpdate({ id: "task-auth", status: "completed" });
// → 自动检查：谁被 task-auth 阻塞？
// → task-pay 解锁，状态变为 ready

// 3. 开始执行
await TaskUpdate({ id: "task-pay", status: "in_progress" });
// → 可以在后台子代理中执行

// 4. 后台执行
const bgSession = agent.spawnBackground({
    task: "task-pay",
    prompt: "重构支付模块...",
});
// → 子代理在独立线程中运行
// → 完成后通知主循环

// 5. 查看状态
const tasks = await TaskList();
// → 返回所有任务及其依赖关系
```

### 3.2 为什么 Codex 和 Pi-Mono 不需要内置任务系统？

```
Codex 的替代方案:
├── 多代理协作 (agent_jobs)
│   → 不需要任务 DAG，直接 spawn 子代理
│   → 子代理有自己的完整生命周期
│   → 父代理等待子代理完成
├── 会话持久化 (SQLite)
│   → 历史不丢失，不需要外部任务文件
└── 记忆系统 (memories)
    → 长期上下文不依赖任务记录

Pi-Mono 的替代方案:
├── 会话树 (session tree)
│   → /fork 创建并行探索分支
│   → /tree 可视化所有分支
│   → 分支本身就是"任务"的物理载体
├── 文件系统 (PLAN.md)
│   → 模型自己创建和更新计划文件
└── 扩展 (Extension API)
    → 如果需要，扩展可以实现任意复杂的任务系统
```

## 四、后台任务（s13）

learn-claude-code 教了守护线程 + 通知队列的模式：

```python
# learn-claude-code s13 的核心思想
class BackgroundTaskRunner:
    def __init__(self):
        self.notification_queue = Queue()  # 线程安全
        self.lock = Lock()
    
    def spawn(self, task_id, func):
        """在后台线程运行"""
        thread = Thread(target=self._run, args=(task_id, func))
        thread.daemon = True
        thread.start()
    
    def _run(self, task_id, func):
        result = func()
        with self.lock:
            self.notification_queue.put((task_id, result))
    
    def drain_notifications(self) -> list:
        """在每次 LLM 调用前清空通知"""
        results = []
        while not self.notification_queue.empty():
            results.append(self.notification_queue.get())
        return results
```

## 五、Cron 调度（s14）

learn-claude-code 教的 Cron 调度器：

```python
class CronScheduler:
    def __init__(self):
        self.schedules = []  # ScheduleRecord 列表
        self.notification_queue = shared_queue
    
    def add(self, cron_expr, prompt, durable=True):
        self.schedules.append(ScheduleRecord(
            cron_expr=cron_expr,
            prompt=prompt,
            last_fired_at=None,
            durable=durable,
        ))
    
    def _checker_loop(self):
        """后台线程，每 60 秒检查一次"""
        while True:
            now = datetime.now()
            for schedule in self.schedules:
                if should_fire(schedule, now):
                    schedule.last_fired_at = now
                    self.notification_queue.put(schedule.prompt)
            sleep(60)
```

### 三个仓库对比

```
Codex:       无内置 cron。但 SDK/app-server 可以外部调度。
Pi-Mono:     无内置 cron。可以用系统 cron + pi CLI 实现。
Claude Code:  有内置 CronCreate/CronDelete/CronList 工具。
              支持 ScheduleWakeup（动态延迟唤醒）。
              → 这是 Claude Code 独有的功能。
```

**Claude Code 的 ScheduleWakeup 是什么？**

```typescript
// Claude Code 独有：动态自我调度
// 模型可以告诉系统"N 秒后叫醒我"

ScheduleWakeup({
    delaySeconds: 270,  // 270 秒后唤醒（在 5 分钟缓存窗口内）
    reason: "等待构建完成",
    prompt: "检查构建是否完成",
});

// 系统在 270 秒后自动向模型发送 prompt
// 模型继续工作

// 关键设计：缓存感知
// - < 300s: prompt cache 还热，唤醒便宜
// - > 300s: cache 失效，唤醒贵但等得更久
// - = 300s: 最差——cache 刚好失效但等得不够久
// → 所以文档说"永远不要选 300s"
```

## 六、总结对比表

| 功能 | learn-claude-code | Codex | Pi-Mono | Claude Code |
|------|------------------|-------|---------|-------------|
| **会话计划** | TodoWrite (s03) | 无 | 无（文件替代） | TaskCreate 系列 |
| **任务持久化** | .tasks/ JSON (s12) | SQLite 会话历史 | JSONL 会话文件 | .tasks/ JSON |
| **任务依赖** | blockedBy DAG (s12) | 无 | 无 | blockedBy DAG |
| **后台执行** | 守护线程 + 队列 (s13) | tokio task + 事件通道 | 无（并行工具执行替代）| AgentTool + React 状态 |
| **定时调度** | Cron 表达式 (s14) | 无 | 无 | CronCreate + ScheduleWakeup |
| **计划模式** | 无 | 无 | 无 | EnterPlanMode / ExitPlanMode |

### 关键洞察

```
Codex:       "我不管你怎么规划，我负责安全地执行"
Pi-Mono:     "规划是用户的事，我提供灵活的工具"  
Claude Code:  "我帮你规划、跟踪、调度，全方位管理"
```

Claude Code 在规划和任务管理上投入最多，因为它的设计目标是**让模型自己管理复杂的多步工作流**。Codex 和 Pi-Mono 则更偏向"执行者"角色。
