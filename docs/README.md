# 阶段文档约定

这个仓库从现在开始执行一个固定约定：

每完成一个阶段，必须补一份“阶段讲解文档”。

目的不是单纯记录功能，而是确保这套代码以后可以用于：

1. 自己回顾实现过程
2. 面试时讲清楚设计取舍
3. 简历上把项目写得具体、可信、可展开

## 每个阶段文档必须包含的内容

1. 阶段目标
2. 本阶段最终交付了什么
3. 目录组织和文件结构
4. 每个文件负责什么
5. 每个文件中的代码是怎么组织的
6. 一次完整请求是如何流过这套代码的
7. 为什么这样拆模块，而不是写成一个大文件
8. 当前实现的边界、限制和下一步扩展点
9. 这一阶段可以如何写进简历
10. 面试时如何口头解释这一阶段

## 文件命名约定

后续阶段统一放在 `docs/` 目录下，命名格式：

```text
phase-01-单Agent核心讲解.md
phase-02-持久化上下文权限记忆讲解.md
phase-03-通道网关与手机接入讲解.md
...
```

## 写作要求

- 解释必须贴着当前仓库真实代码来写
- 不写空泛术语，尽量说清楚“为什么拆成这样”
- 尽量让读者能从文档反推代码结构
- 必须包含可用于简历和面试表达的版本
