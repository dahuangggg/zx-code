# 05 - 高频面试问答

## Q1：请介绍一下这个项目

一句话：

> 这是一个用 Python 从零实现的本地 Coding Agent Runtime，参考 Claude Code 和 Codex 的核心机制，实现了 ReAct 主循环、工具调用、上下文压缩、记忆、权限、多通道、调度、子代理、MCP 和插件系统。

展开版：

> 项目的核心是 `run_task()`，它实现了 ReAct 循环：恢复历史、组装上下文、调用模型、执行工具、回写工具结果，再继续调用模型，直到模型不再调用工具。
>
> 在核心 loop 之外，我做了生产化补齐：ContextGuard 管 token 预算，ResilienceRunner 管模型错误恢复，SessionStore 做 JSONL 持久化，SystemPromptBuilder 做 prompt 管线，PermissionManager 做工具审批，DeliveryQueue 做可靠投递，LaneScheduler 做主对话/子代理/Cron/Heartbeat 的优先级调度。
>
> 技术栈是 Python asyncio + Pydantic v2 + litellm。选择 Python 是为了快速实现和清晰展示 Agent 内部机制，选择 litellm 是为了供应商无关。

## Q2：你的 Agent Loop 是怎么实现的？

一句话：

> `run_task()` 是一个 async ReAct while loop，每轮准备上下文、调用模型、执行工具、写回工具结果，直到没有工具调用。

展开版：

> 一开始从 SessionStore 恢复历史消息，追加用户消息。每轮先经过 ContextGuard 处理 token 预算，然后 ResilienceRunner 调用模型。如果模型返回 tool_calls，就按顺序执行：pre hook、ToolRegistry 权限检查和执行、post hook、把 ToolResult 转成 tool message 写回 session。下一轮模型就能看到工具结果继续推理。如果没有 tool_calls，就返回最终答案。

追问：

**为什么不用 LangChain？**

> 这个项目目标是深入实现 Agent Runtime，而不是快速调库。自己实现 loop、tool registry、context compact、permission、delivery，能解释每个设计取舍，也更适合面试展示底层理解。

## Q3：上下文压缩怎么做？

一句话：

> 两阶段：先截断单条超大工具结果，再在总 token 超预算时总结旧历史并保留最近消息原文。

展开版：

> `ContextGuard.prepare()` 每轮调用前执行。第一步 `truncate_large_tool_results()`，单条 tool message 超过 6000 字符就截断。第二步计算总 token，如果超过预算，就用 `_safe_split_index()` 找到安全切分点，避免 assistant tool call 和 tool result 被拆开。旧消息用 compact model 总结，最近 6 条保留原文。

追问：

**为什么保留最近 6 条？**

> Coding Agent 最需要刚刚读到的文件内容、刚失败的测试、刚执行的编辑结果。这些信息摘要后容易丢细节，所以近场保留原文，远历史才摘要。

## Q4：记忆系统和会话历史有什么区别？

一句话：

> 会话历史记录当前 session 发生了什么；记忆系统保存跨 session 仍然有价值的用户偏好和项目事实。

展开版：

> SessionStore 是 JSONL append-only，每个 session 一个文件，用于恢复对话。MemoryStore 是 Markdown + YAML frontmatter，用于保存 user/feedback/project/reference 类型的长期记忆，并通过 SystemPromptBuilder 注入 prompt。记忆默认不能覆盖源码、测试和当前工具结果，而且 memory_append 默认需要审批。

追问：

**为什么不用向量数据库？**

> 本地 Coding Agent 的长期记忆通常几十到几百条，Markdown 文件足够。向量数据库依赖重、维护成本高。真正瓶颈是记忆质量和写入控制，不是索引性能。

## Q5：System Prompt 怎么组装？

一句话：

> 用 `SystemPromptBuilder` 按 section 管线组装，稳定内容放前面，动态内容放后面。

展开版：

> section 顺序是 Identity、Operating Rules、Tools、Project、Project Instructions、Skills、Memory、Runtime、Tasks、Todos。工具、项目指令、技能索引比较稳定，放前面有利于 prompt cache；Runtime、Tasks、Todos 经常变化，放后面降低缓存失效范围。

追问：

**Skills 为什么只注入索引？**

> 技能全文可能很长，不应该每轮占用 token。prompt 里只放名称和描述，模型需要时调用 `load_skill` 工具加载完整 Markdown。

## Q6：工具系统怎么保证参数正确？

一句话：

> 每个工具用 Pydantic input model 定义参数，`model_json_schema()` 生成给 LLM 的 schema，执行时 `model_validate()` 验证参数。

展开版：

> 工具继承 `Tool`，声明 `name`、`description`、`input_model`，实现 `run()`。ToolRegistry 执行时先权限检查，再执行工具。参数错误会被捕获为 `ToolResult(is_error=True)`，不会让主循环崩溃。

追问：

**工具异常为什么回写给模型？**

> 因为工具失败是 Agent 正常推理的一部分。比如文件不存在、测试失败、权限拒绝，模型看到错误后可以换路径、修复代码或请求用户确认。

## Q7：权限系统怎么设计？

一句话：

> 三层权限：工具级策略、配置文件规则、内置安全兜底，返回 allow/deny/ask 三态。

展开版：

> `PermissionManager.decide()` 先看 `tool_policies`，再看 `.zx-code/permissions.toml` 中的 fnmatch 规则，最后看内置规则，比如 bash 里的 `rm -rf`、`sudo`、`dd`，以及 write/edit 的敏感路径和工作目录越界。`allow` 直接执行，`deny` 返回错误，`ask` 走 approval callback。

追问：

**和 Codex 沙箱比呢？**

> Codex 是 OS 级沙箱，本项目是应用层权限。它能展示权限决策链和审批机制，但不能替代 Seatbelt/Landlock。生产级安全可以把工具执行再放进容器或系统沙箱。

## Q8：LLM API 失败怎么恢复？

一句话：

> 先分类，再按类型恢复，并用 RecoveryBudget 防止无限重试。

展开版：

> `classify_error()` 把异常分为 rate_limit、overflow、max_tokens、timeout、auth、billing、unknown。rate_limit 走指数退避，overflow 调 ContextGuard compact 后重试，max_tokens 追加“请继续”消息，timeout/auth/billing 不盲目重试。FallbackModelClient 还会对失败 profile 做 cooldown，然后尝试下一个模型 profile。

追问：

**为什么字符串匹配异常？**

> litellm 支持很多供应商，异常类不统一，但错误消息关键词比较稳定，比如 429、context_length、billing。这里是供应商无关架构的现实取舍。

## Q9：如何保证消息不丢？

一句话：

> DeliveryQueue 使用文件系统 WAL 思想，先写 queued 文件，再发送，成功移到 sent，失败记录 retry。

展开版：

> `enqueue()` 会把 DeliveryEntry 原子写入 `queued/`，写入流程是 tmp file、flush、fsync、os.replace、fsync directory。DeliveryRunner 发送消息，成功 mark_sent，失败 mark_retry 并设置 next_retry_at。DeliveryDaemon 后台周期性重试，超过最大次数进入 failed。

追问：

**能保证 exactly-once 吗？**

> 不能。外部平台 API 不是事务资源。如果发送成功但进程在 mark_sent 前崩溃，可能重复发送。这里保证的是 at-least-once 倾向和崩溃后可恢复。要 exactly-once 需要平台支持幂等 key 或业务侧去重。

## Q10：LaneScheduler 为什么是单 worker？

一句话：

> 为了保证同一用户消息有序，并让主对话优先于子代理、Cron 和 Heartbeat。

展开版：

> 所有 agent turn 进入 PriorityQueue，优先级是 main=0、subagent=10、cron=20、heartbeat=30。单 worker 取出执行，避免同一用户多条消息并发乱序。ContextVar 检测如果已经在 worker 内部，再次调用 scheduler 就内联执行，避免主 agent 等子代理、子代理又排队等同一个 worker 的死锁。

追问：

**吞吐会不会差？**

> 对单机 Coding Agent 可以接受，因为瓶颈主要是 LLM 和工具副作用，不是 CPU。如果要扩展多用户，可以按 channel、account 或用户分片多个 scheduler。

## Q11：子代理怎么隔离上下文？

一句话：

> 每个子代理生成独立 session_id，并通过 `_run_agent_text()` 重新构建 runtime。

展开版：

> `SubagentRunner.run()` 检查递归深度，生成 `<parent>:subagent:<label>:<uuid>` 的 child session id，然后走 subagent lane 执行 `_run_agent_text()`。由于 session_id 不同，SessionStore 中的消息历史独立，父代理只拿最终 `SubagentRunResult`。

追问：

**子代理会不会无限递归？**

> 不会，`current_depth` 每次 +1，达到 `subagent_max_depth` 会抛 `SubagentRecursionError`。

## Q12：MCP 是怎么接入的？

一句话：

> MCPToolRouter 发现外部 MCP server 的工具后，把它包装成 MCPProxyTool 注册进 ToolRegistry。

展开版：

> `_attach_mcp_tools()` 创建 `StdioMCPClient`，通过官方 MCP SDK 连接 server，`discover_tools()` 返回工具列表。每个工具被命名为 `mcp__server__tool`，schema 来自 MCP server 的 input_schema。注册进 ToolRegistry 后，模型像调用内置工具一样调用 MCP 工具，也走统一权限和错误包装。

追问：

**为什么不用自己实现 MCP 协议？**

> MCP 包含 stdio transport、JSON-RPC、initialize、list_tools、call_tool 等协议细节。官方 SDK 兼容性更好。本项目关注的是如何把 MCP 工具接入 Agent 工具体系，而不是重复造协议栈。

## Q13：这个项目最有技术含量的地方是什么？

可以从三个层次回答：

1. Agent 核心机制：ReAct loop、tool call、context compact、prompt pipeline。
2. 工程化可靠性：权限、错误恢复、session 持久化、可靠投递、调度。
3. 扩展能力：Skills、Memory、MCP、Plugins、Subagent、Team、Worktree。

推荐回答：

> 我认为不是某一个算法最难，而是把 Agent 从 demo 变成 runtime 的边界设计：核心 loop 不依赖通道；工具系统不依赖具体工具；模型层不绑定供应商；状态都可持久化；高风险操作有权限和 hook；长任务有上下文压缩和恢复。这些组合起来才像一个可运行的 Coding Agent 框架。

## Q14：项目有什么不足？

诚实回答：

> 第一，安全上没有 OS 级沙箱，只是应用层权限。第二，文件系统持久化适合单机，不适合多机高并发。第三，UI 不是 Claude Code 那种完整 TUI。第四，记忆系统偏显式写入，没有复杂的自动提取和去重。第五，工具并发执行还比较保守，主要是串行。

补充：

> 这些是有意取舍。项目目标是用较少依赖实现核心机制，并能清楚讲解。如果继续做生产化，下一步是工具执行沙箱、多 scheduler 分片、真实 embedding 记忆检索和更完整的交互 UI。
